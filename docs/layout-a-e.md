## 1. 背景与问题

V2 落地后，用户反馈与代码现状暴露同一问题：

| 现象 | 根因（代码 / 结构） |
|------|---------------------|
| 待办、已完成越多，页面越长 | 右栏 `battle-stage` 整页延展，无内部滚动上限 |
| AI 对话区「越来越小」 | `.chat-messages { max-height: 320px }` 写死上限 |
| 左右视觉失衡 | `align-items: start`，左栏固定 ~420px，右栏无限增高 |
| 已完成抢战场注意力 | 进行中 + 已完成同列堆叠，都在主视野竞争 |

产品叙事是 **「左对话主舞台 + 右待办副舞台」**，当前布局在任务量上升后变成 **「右清单主、左聊天副」**，与气质不符。

---

## 2. 方案概述（A + E）

### 方案 A — 双栏等高 + 各自内部滚动

桌面端（≥840px）两栏 **sticky 钉在视口内**，页面主体高度不再随任务数增长；长内容在栏内滚动。

### 方案 E — 已完成移出战场，进「奇迹纪念馆」

右栏 **只承载进行中**；已完成从战场区移除，归档至 HUD 展开区（或等效入口）。

### 组合效果

```
┌─────────────────────────────────────────────────────┐
│ Header                                              │
├──────────────────────┬──────────────────────────────┤
│ AI 对话区 [sticky]    │ 待办战场 [sticky]             │
│ ┌──────────────────┐ │ ┌──────────────────────────┐ │
│ │ Mascot           │ │ │ 进行中 (N)  [内部滚动]    │ │
│ │ 气泡 / 战报 ↕     │ │ │ 全部未完成任务            │ │
│ │ （无 320px 上限）  │ │ │ 按腐败度排序              │ │
│ ├──────────────────┤ │ └──────────────────────────┘ │
│ │ 输入框 [sticky底] │ │                              │
│ └──────────────────┘ │                              │
├──────────────────────┴──────────────────────────────┤
│ HUD · 展开 → 奇迹纪念馆 (M) / 摆烂报告 / 导入导出      │
└─────────────────────────────────────────────────────┘
```

## 3. 设计原则

1. **战场 = 当前要对付的关卡** — 仅 `completed === false` 的任务
2. **纪念馆 = 已发生的奇迹** — 仅 `completed === true`，按完成时间倒序
3. **对话 = 抬杠 + 战报时间流** — 左栏随视口变高，战报不再挤在 320px 里
4. **先完成的不会丢** — 归档到纪念馆，与进行中列表互不抢位
5. **不隐藏进行中任务** — 全部显示在战场内滚动区，**不做「只露 5 条」硬上限**

## 4. 详细规格

### 4.1 桌面布局（≥840px）

| 项 | 规格 |
|----|------|
| 网格 | 保持 `45fr / 55fr` 或微调为 `48fr / 52fr`（实现时 A/B） |
| 栏高 | `height: calc(100vh - header - hud - pagePadding)`，`position: sticky; top: headerHeight` |
| 左栏 | `display: flex; flex-direction: column; min-height: 0` |
| 消息区 | `flex: 1; overflow-y: auto`，**移除** `max-height: 320px` |
| 输入区 | `composer-form` 在左栏底部，`flex-shrink: 0` |
| 右栏 | 同上；列表区 `flex: 1; overflow-y: auto` |

### 4.2 进行中列表（战场）

| 项 | 规格 |
|----|------|
| 数据 | 全部 `!task.completed` |
| 排序 | 默认 **腐败度降序**（`rot.level` → `ageHours`），最烂最上 |
| 滚动 | 任务再多也在栏内滚，不撑高页面 |
| 提示 | 当 `N > 8` 时，列表顶显示：「{N} 项进行中 · 越烂越靠上」 |
| 空状态 | 沿用 `BattleEmpty` |

**明确不做：** ~~仅展示 5 条，其余隐藏~~（易误解为丢任务，已否决）

### 4.3 奇迹纪念馆（已完成）

| 项 | 规格 |
|----|------|
| 入口 | `HudBar` 展开区：「💀 奇迹纪念馆 (M)」按钮 → 弹层 / 侧滑 / 展开列表 |
| 数据 | 全部 `task.completed`，按 `completedAt` **降序**（刚完成在上） |
| 单条展示 | 任务名、逃跑次数、`completionRoast`、可选「当初抬杠」摘要 |
| 操作 | 与现 HUD 一致：清空已完成、导出仍包含已完成数据 |
| 战场 | **不再渲染** `done-block` / `task-list-done` |

### 4.4 完成动线（与 V2 战报衔接）

```
用户抓住「✓ 完成打卡」
  → TaskItem 1.6s completion banner（不变）
  → 左栏战报气泡（V2 已有）
  → persist：completed + completionRoast + completedAt
  → 卡片从战场列表移除（不再出现在右栏）
  → 纪念馆列表头部出现新条目
  → HUD 完成 +1
```

### 4.5 移动端（<840px）

| 项 | 规格 |
|----|------|
| Tab | 保持「抬杠 \| 待办」 |
| 待办 Tab | 仅进行中 + 内部滚动；顶部可显示「纪念馆 M 条 →」链到 HUD 或独立 Tab 入口 |
| 纪念馆 | 优先放 HUD 展开，避免三 Tab；若后续任务量极大再考虑第三 Tab |
| 对话区 | 当前 Tab 内 `flex` 撑满，去掉 320px 上限 |

### 4.6 CSS 变更清单（预估）

| 选择器 / 文件 | 变更 |
|---------------|------|
| `.chat-messages` | 删除 `max-height: 320px`，改 `flex: 1; min-height: 0; overflow-y: auto` |
| `.stage-grid` | 桌面：`align-items: stretch`；子栏 sticky + 固定高度 |
| `.chat-stage` | `min-height` 改为与视口计算挂钩，非固定 420px |
| `.battle-stage` | 列表容器可滚动；移除 `.done-block` 相关 DOM |
| `.page` | 桌面 `overflow: hidden` 或限制主区高度，防止双滚动条打架 |
| `HudBar.tsx` | 新增纪念馆 UI |
| `App.tsx` | 已完成数据改从 HUD/纪念馆读；战场只 map activeTasks |

## 5. 用户场景验证

### 场景 1：先完成旧的，再做新的

1. 用户有 A、B、C 进行中，先完成 A  
2. 战场剩 B、C；纪念馆 1 条（A，带评语）  
3. 新加 D，战场 B、C、D — **A 不占战场位**

### 场景 2：大量进行中

1. 用户 15 项均未完成  
2. 战场列表 15 条，栏内滚动，腐败最高的在顶部  
3. 左栏对话/战报高度不受右栏任务数影响  

### 场景 3：战报增多

1. 多次甩锅 / 完成产生战报  
2. 左栏消息区随 sticky 栏变高，内部滚动查看历史战报

## 8. 成功指标（定性）

- 桌面 10+ 进行中时，用户无需滚整页即可在左栏看完最近战报

- 新用户理解：「右边打怪，左边听骂，下面查账」
